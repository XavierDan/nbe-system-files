Section 1 — Product Spec
Assumptions
1. NBE is a single-page web application hosted on EmbarkMedia.com. No user accounts or authentication are required at v0.5.
2. Input is plain text. Bracket annotations (e.g., [BEAT: Arrival]) are parsed if present, but the system does not require them.
3. The primary output unit is a rendered card — one card per identified beat, displayed in sequence in the browser.
4. "Copy-ready" means the user can select and copy card content directly from the browser. A dedicated copy-to-clipboard button is deferred.
5. Brand token application (colors, typography) is applied at the card rendering layer, not at the API response layer.
6. V1 scope excludes user accounts, saved sessions, PDF or DOCX export, and sharing links.
________________


What NBE Is
The Narrative Blueprint Engine is an AI-powered creative workspace. A professional video or podcast creator pastes raw script content — a production script, VO draft, dialogue transcript, or paper edit — into a single input field. NBE processes that content through SM², its internal semantic logic layer, and returns a structured sensory production blueprint rendered as a sequence of in-browser cards.
NBE is not a search tool, a stock footage finder, or a shot list generator in the conventional sense. It is a semantic translation system: it reads what a script is doing emotionally and dramatically, then produces specific, rationale-backed production direction across visual and audio registers simultaneously.
The workspace serves two narrative formats equally: video production and audio-only narratives (podcast, documentary audio). The same input field and the same output structure serve both. The distinction is handled inside SM².
________________


Inputs
Primary input field — Script Content
* Accepts: production scripts, VO drafts, dialogue transcripts, paper edits
* Format: plain text, pasted directly
* Bracket annotations supported but not required (e.g., [SCENE: Opening], [VO], [BEAT: Grief])
* No minimum or maximum word count enforced at v0.5; practical range is one paragraph to approximately 2,000 words per session
* No file upload at v0.5; paste-only
Secondary input field — Project Context (optional)
* A short free-text field for tone, audience, format, and distribution context
* Examples: "Documentary. Intimate tone. PBS audience." / "Podcast intro. High energy. Gen Z."
* Used by SM² to calibrate the register and the specificity of output
* If left blank, SM² defaults to general video narrative assumptions
________________


Outputs
NBE produces a sequential set of rendered cards — one per identified beat — displayed in the browser immediately below the input area upon generation. Cards render in order, top to bottom, matching the narrative sequence of the input.
The rendered output is the product. There is no intermediate state visible to the user. The JSON produced by SM² is consumed entirely by the rendering layer; the user never sees raw JSON.
________________


The Rendered Card — V0.5 Deliverable
A rendered card is the primary user-facing representation of one SM² output unit. Each card corresponds to a single identified beat in the input.
Card fields and hierarchy:

│     Example: "The Decision" / "Loss Arrives" / "System Failure Revealed"
│
├── Beat Text                     [string]
│     The source script passage this card corresponds to.
│     Displayed verbatim, pulled from input.
│
├── Visual Directions             [object]
│   ├── Literal                   [string]
│   │     Direct, on-the-nose visual representation.
│   │     Specific enough to brief a DP or stock researcher.
│   │
│   ├── Interpretive-Metaphorical [string]
│   │     A visual that carries the meaning without stating it.
│   │     Paired with a rationale sentence explaining the match.
│   │
│   └── Tonal-Atmospheric        [string]
│         A visual that establishes emotional register and
│         texture rather than narrative content. Mood, light,

Display logic:
* Cards render sequentially, top to bottom, in narrative order
* Each card is visually bounded — a distinct block with clear separation from adjacent cards
* Beat Label renders as the card header in Playfair Display
* Visual and Audio sections render as labeled subsections using IBM Plex Mono labels
* Body text renders in Karla at the appropriate weight
* EmbarkMedia brand tokens applied: burnt orange (#CC5500) for labels and accents, white card background, near-black body text
* Cards are selectable and copyable in the browser natively; no custom copy mechanism required at v0.5
Card purpose: Each card is a self-contained creative brief for a single narrative moment. A producer should be able to hand a single card to a DP, sound designer, or music supervisor and receive a meaningful response without additional context.

Section 2 — Logic-Layer Spec
Assumptions
1. SM² is implemented entirely through a structured system prompt passed to the Claude API (Sonnet model) via a PHP proxy. There is no separate ML model, classifier, or custom fine-tuning at v0.5.
2. Beat identification is performed by the language model in a single API call. There is no pre-processing pass.
3. The JSON schema below is a draft assumption, not a validated contract. It must be tested against real input before implementation begins.
4. Semantic matching is performed by the LLM based on narrative function and emotional weight. There is no keyword lookup table or rules engine.
5. For ambiguous or abstract beats, SM² applies a defined fallback behavior (specified below) rather than returning null or empty fields.
6. The same system prompt governs both video and audio-only narrative formats. Format-specific behavior is triggered by the optional project context field.
________________


What SM² Is
SM² (Semantic Multi-Sensory Matching Logic) is the internal logic layer that powers NBE. It is not a separate product and is never referenced in user-facing copy. SM² is the interpretive system underneath the workspace interface.
SM² performs one core operation: it takes a unit of script content and matches it to sensory production directions across multiple registers simultaneously, based on the semantic content of that unit — its narrative function, emotional weight, and meaning — not its surface keywords or tags.
SM² reads text the way an experienced editor reads a script: not word by word, but for what the moment is doing, what it needs to carry, and what a viewer or listener needs to feel.
________________


The Beat — Minimum Unit of Semantic Content
Definition: A beat is the minimum unit of semantic content that SM² processes as a single, coherent unit of meaning. A beat is a passage of script content that serves a single identifiable narrative or emotional function.
Beat boundaries are determined by:
* Shift in narrative function (e.g., from establishing context to introducing conflict)
* Shift in emotional register (e.g., from tension to release)
* Shift in subject or speaker that changes the dramatic focus
* Natural paragraph or scene breaks in the source text
* Explicit bracket annotations in the input (e.g., [BEAT], [SCENE]) — these are honored as boundary markers when present
What a beat is not:
* A single sentence (unless that sentence performs a complete, discrete narrative function)
* A topic paragraph in expository writing (unless the topic shift constitutes a genuine emotional or narrative turn)
* An arbitrary chunk of a fixed word count
Beat granularity: SM² should bias toward fewer, larger beats rather than many small ones. A four-minute script should produce approximately four to eight beats, not twenty. Over-segmentation fragments the sensory direction and reduces usability. Under-segmentation loses the nuance of the narrative arc.
Beat label generation: Each beat is labeled with a concise, evocative descriptor generated by SM². Labels should be two to five words, expressive rather than procedural. "The Decision," not "Decision Point." "Loss Arrives" not "Emotional Beat 3."
________________


Fallback Behavior for Ambiguous Beats
Some beats are highly abstract, conversational, or expository — they carry no obvious visual anchor and resist metaphor. SM² must resolve these without returning empty or generic output.
Fallback logic for ambiguous beats:
1. Identify the emotional register. Even purely informational content has a tone. A calm explanation and a tense disclaimer occupy different emotional spaces. SM² uses a register as the entry point when the semantic content is thin.

2. Anchor to the narrative function, not the content. If a beat is purely transitional — moving the listener from one section to the next — its function is transition. SM² maps transition as a sensory category: visual fade, threshold imagery, sonic crossfade behavior.

3. Use the surrounding beats as context. SM² reads the incoming beat in sequence, not isolation. A beat that is ambiguous on its own gains meaning from what precedes and follows it. SM² should reference that context explicitly in its rationale.

4. Default to Tonal-Atmospheric as the primary lane. When Literal and Interpretive-Metaphorical are difficult to justify, Tonal-Atmospheric can carry the card. A beat without a clear visual subject still has a temperature, a texture, a light quality.

5. Never return a null or placeholder. All five output fields (Literal, Interpretive-Metaphorical, Tonal-Atmospheric, Sound Design, Music) must be populated for every beat. If the match is a stretch, the rationale field should say so — "This beat resists direct visual anchoring; the Tonal-Atmospheric direction is the primary recommendation."

SM² Quality Scoring Rubric
Each SM² output is evaluated at the beat level across five dimensions. Each dimension is scored from 0 to 3.
Scoring scale:
   * 0 — Failing. Output is absent, generic, or unusable.
   * 1 — Marginal. Output is present but lacks specificity or actionability.
   * 2 — Acceptable. Output is specific and usable but not distinguished.
   * 3 — Excellent. Output is specific, well-matched, and could be handed directly to a collaborator.
________________


Dimension 1 — Beat Identification Does SM² identify beats at the correct level of granularity for this input?
   * 0: Beats are wrong — too granular (one sentence each), too broad (entire script as one beat), or misaligned with actual narrative turns.
   * 1: Beat count is in the right range, but boundaries are arbitrary rather than semantically justified.
   * 2: Beats correspond to genuine narrative or emotional turns. Minor misalignment at one boundary.
   * 3: Beat segmentation is precise, justified, and would match what an experienced editor would identify.
________________


Dimension 2 — Visual Directions: Specificity and Actionability. Are the three visual lane directions sufficiently specific to brief a DP or a stock researcher?
   * 0: Directions are generic ("a person looking sad," "a city skyline").
   * 1: Directions have some specificity but rely on stock-footage clichés or under-describe the shot.
   * 2: Directions are specific and actionable. A researcher could source based on the description.
   * 3: Directions are specific, unexpected, and paired with a rationale that makes the semantic match clear.
Passing threshold for this dimension: 2 or above on all three lanes per beat.
________________


Dimension 3 — Visual Rationale Quality: Does the rationale explain why this specific image matches this specific beat?
   * 0: Rationale is absent or restates the direction without explanation.
   * 1: Rationale is present but generic ("this image conveys emotion").
   * 2: Rationale makes a clear connection between the image and the beat's meaning.
   * 3: Rationale is precise and reveals a non-obvious semantic logic that strengthens the direction.
________________


Dimension 4 — Audio Directions: Sound Design Is the sound design direction specific, textured, and emotionally functional?
   * 0: Generic ("ambient noise," "background music").
   * 1: Some specificity but missing source, quality, or emotional function.
   * 2: Source, quality, and function are all described. Usable as a brief.
   * 3: Direction is specific, unexpected, and would produce a meaningfully different result than a generic choice.
________________


Dimension 5 — Audio Directions: Music. Is the music direction specific, and does it include a licensing tier recommendation?
   * 0: Generic ("sad music," “upbeat track”). No licensing guidance.
   * 1: Some specificity but missing instrumentation, tempo, or dynamics. Licensing tier present.
   * 2: Instrumentation, tempo, dynamics, and licensing tier all present. Usable as a brief.
   * 3: Direction is specific enough to guide a music supervisor or composer. Licensing tier is appropriate to the context.
________________


Overall passing threshold:
   * Average score of 2.0 or above across all five dimensions per beat.
   * No dimension scored 0 on more than one beat in the test set.
   * All five output fields (three visual lanes, two audio registers) are populated for every beat in every test run.
________________


PHP Proxy Acceptance Criteria
The PHP proxy performs one function: it receives a POST request from the NBE frontend containing the SM² payload, forwards it to the Claude API with the appropriate headers and system prompt, and streams the JSON response back to the browser.
Functional requirements — all must pass:
   * [ ] Accepts POST request from frontend with Content-Type: application/json
   * [ ] Injects API key server-side (never exposed to frontend)
   * [ ] Attaches system prompt to API call correctly
   * [ ] Forwards script_content and project_context fields from frontend payload to API
   * [ ] Streams response back to browser (streaming mode preferred; polling acceptable at v0.5)
   * [ ] Returns a valid JSON object matching the SM² output schema on success
   * [ ] Returns a structured error response on API failure — not a raw error or a blank screen
   * [ ] Handles empty or whitespace-only input gracefully — rejects before sending to API
Latency targets (shared hosting, Claude Sonnet):
Metric
	Target
	Acceptable Upper Bound
	Time to first token (streaming)
	< 3 seconds
	< 5 seconds
	Full response for a 4-beat output
	< 20 seconds
	< 30 seconds

	45 seconds

	Error response time on API failure

Week 1 — What Is a Spec Prompt?
What it is
A spec prompt is a structured document written in natural language — not code — that tells an AI model what to build, how to reason about it, and what the output should look like. It is not a chat message. It is closer to a design brief or a technical requirement document, except the "engineer" reading it is a language model.
A spec prompt defines the system's behavior before a single line of code is written. It is the architectural layer above the code.
Why it matters
Without a spec prompt, you are vibe-coding — sending individual messages and hoping the output accumulates into something coherent. Vibe-coding works for exploration. It does not work for building something another person (or another AI instance) can pick up and extend.
A spec prompt makes the system repeatable and transferable. It is also the first artifact that a real software engineer can read. The spec prompt is where a prompt-driven project transitions from a personal creative tool to something with structural integrity.
What we did
We produced Master Specification Prompt v0.1, then revised it to v0.2 based on feedback from Gemini and Perplexity. The prompt instructs Claude to generate five standalone sections — Product Spec, Logic-Layer Spec, Test Plan, Roadmap, and Learning Curriculum — in a single run. We learned that a well-constructed spec prompt can produce document-quality output that functions as real working material, not just a starting point for editing. We also learned that getting feedback on a prompt from other AI platforms before running it is a meaningful quality-control step.
________________


Week 2 — What Is a Logic Layer?
What it is
A logic layer is the part of a system that does the thinking — the rules, interpretations, and decisions that happen between the raw input and the final output. In a traditional software system, this might be a rules engine, a classifier, or a database query. In NBE, the logic layer is SM²: a set of instructions that tells the language model how to read a script and translate its meaning into a direction for sensory production.
The logic layer is what separates a tool from a system. A tool takes input and produces output. A system has a defined way of reasoning about that input before producing output.
Why it matters
Without a named, defined logic layer, the system's behavior is invisible — it works until it doesn't, and you don't know why either happened. Naming and defining SM² enables us to describe its behavior, specify its failures, and deliberately improve it. It also means a software engineer can eventually implement it properly — replacing the prompt-driven version with a more robust architecture when the time comes.
What we did
We named the logic layer SM² — Semantic Multi-Sensory Matching Logic — and defined what it does: reads semantic content (narrative function, emotional weight, meaning) and maps it to sensory production directions across five output registers simultaneously. We defined the beat as the minimum unit of input, specified fallback behavior for ambiguous content, and drafted a JSON schema for the input/output contract. We also established a critical distinction: NBE is the workspace the user interacts with. SM² is what runs underneath it. That distinction is load-bearing — it keeps the product thinking and the engineering thinking separate, which matters as the project scales.

Open Risks
The following items are unresolved as of this draft. Each should be validated before implementation begins or before the relevant phase gate is cleared.
1 — JSON Schema (draft only) The SM² output schema has not been validated against real API output. The Claude Sonnet model may not reliably produce JSON that conforms to the schema without additional prompt engineering. Validation required: run the system prompt, parse the JSON output, and confirm schema conformance before building the rendering layer.
2 — Quality Scoring Rubric (proposed, not field-tested) The five-dimensional rubric has not been applied to real SM² output. Scores are calibrated on assumed best and worst cases, not actual output. Required: run the test set, score the output, and revise the rubric if the calibration is off. A rubric that passes everything or fails everything is not useful.
3 — Beat Boundary Detection on Messy Real-World Input The beat definition assumes reasonably coherent input. Real-world transcripts contain interruptions, overlapping timecodes, embedded editor chatter, missing brackets, and corrupted speaker labels (as documented in the SCRIPT_TRAINING_EXAMPLES file). SM² behavior on genuinely messy input has not been tested. Required: include one messy transcript in the v0.5 test set and evaluate specifically for beat identification failures.
4 — PHP Proxy Latency Targets (assumed, not benchmarked) The latency targets (< 3 seconds to first token, < 20 seconds full response for a 4-beat output) are based on typical Claude Sonnet API behavior and not on actual measurements from the target shared hosting environment. Shared hosting adds unpredictable overhead. Required: benchmark during the Phase B deployment test and revise targets if necessary.
5 — Quality Rubric Calibration Against Real Output The rubric was written before SM² output was formally evaluated. The passing threshold (an average of 2.0 across five dimensions) may be too strict or too permissive, depending on what the system actually produces. Required: calibrate after the first full test run and adjust thresholds before treating any score as meaningful.
6 — Input/Output Schema Validation The JSON schema is a draft contract. The rendering layer must be built against a validated schema — one that has been confirmed through actual API calls, not assumed. Required: Confirm the schema before writing card rendering code. Building a renderer against an unvalidated schema risks significant rework.
7 — System Prompt Scope Creep As the system prompt grows more complex, it may begin to produce inconsistent output — some beats well-matched, others drifting toward generic language. This is a known failure mode of increasingly complex prompts. Monitor for it during Phase A validation. If it appears, the mitigation is modularization: split the system prompt into a core logic prompt and separate register-specific prompts, chained in sequence.
8 — Audio-Only Narrative Parity NBE is spec'd to serve video and audio-only narratives equally. The example input/output pair in Section 2 uses a documentary/video frame. The audio-only use case (podcast, narrative audio) has not been tested with SM². Required: include a podcast monologue in the v0.5 test set and confirm audio directions are meaningfully different and specifically calibrated for audio-only production — not simply video directions with the visual layer deprioritized.
